### Option B ‚Äî original design (kept for reference)

The aim is to keep the "active set" of outcome names warm so cold-start latency doesn't hit the operator. Sketch:

1. **Source of names.** Reuse `buildMarketPriceWatchlist` (already exists in `src/lib/server/marketPrices/watchlist.ts`) ‚Äî it pulls active plan outcomes, saved-combination outcomes, open candidates, inventory items, and execution results into a deduped name list.
2. **Scheduling.** Two reasonable options:
   - **Bun-side cron** in a long-running script (`tools/refresh-prices.ts` already exists for one-shot runs ‚Äî wrap it in a `setInterval` invoked by a separate `bun run refresh-prices:loop` command).
   - **OS-level scheduler** (Windows Task Scheduler / cron) running `bun tools/refresh-prices.ts` on a schedule. Lower coupling, easier to disable.
3. **Cadence.** Daily (24h) covers freshness without burning rate budget. Active plan outcomes are typically 5‚Äì25 names; saved combos add another 50‚Äì200; inventory adds tens. Total ~100‚Äì300 names. At 3s rate-limit floor that's 5‚Äì15 minutes per sweep ‚Äî fine for an off-hours cron.
4. **Calc interaction.** No change to option A's logic. Background sweep just keeps the cache warm; on-demand refresh becomes a no-op for warmed names (already < 30 min old ‚Üí skipped).
5. **Operator visibility.** `/market-prices` already shows freshness + last observed time. A "last sweep at HH:MM" indicator + a "Run sweep now" button would close the loop.
6. **Implementation pointers.**
   - Don't add a new SvelteKit hook; keep server-side "always-on" loops out of the SvelteKit process. Use a separate Bun script.
   - Persist sweep summaries to a small `MarketPriceSweep` table or a JSON log so the UI can display history.
   - Honor the same 30-min skip in `refreshSteamMarketWatchlist` for free ‚Äî running the sweep doesn't cause double-fetching with calc auto-refresh.
   - Optional: per-name priority. Active baskets > active plans > saved combos > inventory > execution history. So the high-value names refresh first if a sweep gets cut short.

Estimated effort: 1‚Äì2 hours. Add when option A's cold-start latency becomes tiresome in real use.

## Confirmed bugs

### B1 ‚Äî Calculator throws `TypeError: r.price.trim is not a function` ‚úÖ ROOT-CAUSED

**Severity:** high. **Confidence:** high. **Fix size:** ~5 lines.

- **Where:** `src/routes/calculator/+page.svelte:67, 96, 98, 157, 159, 173, 180, 181`.
- **Why:** Rows are typed `{ price: string; floatValue: string; ... }`, but `<Input type="number" bind:value={row.price} />` (lines 350‚Äì362) routes through `src/lib/components/Input.svelte:8` which accepts `value?: string | number`. Svelte's `bind:value` on a `type="number"` input coerces to `number`. As soon as the user types a digit, `row.price` becomes a number, and `r.price.trim()` blows up.
- **Fix options:**
  1. Wrap each `.trim()` site: `String(r.price ?? '').trim()`. Minimal blast radius.
  2. Stop using `<Input>` for numeric fields and bind to a string-typed local; coerce at submit. Cleanest.
  3. Make `Input` accept a `kind="text" | "number"` prop and never coerce. Most invasive.
- **Recommendation:** option 1 for the immediate fix, option 2 if we want it permanently safe.
- **Related sites that may have the same bug:** any other component using `<Input type="number">` and treating the bound value as a string. Worth grepping after the calculator fix.

### B2 ‚Äî Steam inventory linking "doesn't seem to work" üü° LIKELY ROOT-CAUSED

**Severity:** high (whole feature perceived broken). **Confidence:** medium (needs runtime confirmation). **Fix size:** small‚Äìmedium.

- **Where:** `src/lib/server/inventory/steamLinkService.ts:84`.
- **The matching code is mostly correct.** Float epsilon is sane, FIFO fallback is reasonable, transactional updates work.
- **Most likely real cause:** Local rows are bucketed by **exact** `marketHashName` string (line 84). Steam's payload may differ subtly from what the user typed manually:
  - StatTrak `‚Ñ¢` unicode vs `(TM)` or omitted
  - Trailing/leading whitespace
  - Unicode normalization form (NFC vs NFD)
  - Capitalization
  - Non-breaking space vs regular space
- The catalog code already has `normalizeMarketHashLookup` (`src/lib/server/utils/marketHash.ts:52`) ‚Äî **the linker doesn't use it.** That's the asymmetry.
- **UI hides the failure mode.** `src/routes/inventory/+page.svelte:142` shows "Synced 0 new links" with no breakdown of which Steam items had no local match (the data exists in `unlinkedSteamItems` but isn't surfaced).
- **Recommended fix path:**
  1. Apply `normalizeMarketHashLookup` to the bucket key in `steamLinkService.ts:84` (and again on the Steam side at line 105). One change, both sides match.
  2. Surface `unlinkedSteamItems` and `missingFromSteam` counts/samples in the UI banner so the user sees *why* nothing linked.
  3. Add the FLOAT_EXACT/BACKFILL/FIFO unit tests that PROGRESS already calls for.
- **Other smell, not root cause:** the Prisma query at `:60‚Äì73` uses `OR: [{ steamAssetId: null }, { steamAssetId: { not: null } }]` ‚Äî a tautology. Dead intent; clean up.

#### I-T1-001 ‚Äî Partition `itemCollectionEV` ranks without wear-proportion ‚úÖ FIXED
- **Where:** `src/lib/server/tradeups/planner/partition.ts:236‚Äì247`.

### Tier 2 ‚Äî Data integrity

#### I-T2-001 ‚Äî Discovery `loadQueueSlotsByPlan` is hardcoded to 10 per plan (medium)
- **Where:** `src/lib/server/discovery/watchlistService.ts:133‚Äì139`.
- **What:** This function ostensibly returns "queue slots by plan," but it just returns `10` for every active plan. The value flows into `rulePriority(rule, planDemand)` at line 196 (`planDemand * 100 + ...`), so every active plan gets a +1000 priority floor regardless of actual planner queue load.
- **Effect:** Discovery target priority is supposed to be demand-driven; the queue-slots component is effectively a constant. Plans with empty queues vs plans with actual planner needs rank identically on the queue-slots axis.
- **Fix:** either query real queue depth from the planner pool, or rip the field out and document priority as basket-only.
